# Tag 体系管理规范

## 设计原则

Tag 采用**扁平列表**，不分层级，自然生长。每条记录可以打多个 Tag。

## Tag 匹配流程

每条新记录需要打 Tag 时：

1. **读取** `$DIARY_HOME/tags.md` 中的现有 Tag 列表
2. **语义匹配**：从列表中选出最贴合的 Tag，可以选多个（通常 1-3 个）
3. **无匹配则新增**：如果现有 Tag 都不合适，新增一个
   - 新增前检查是否已有语义相近的 Tag（如"吃饭"和"饮食"，选已有的那个）
   - 确认需要新增后，追加到 `$DIARY_HOME/tags.md` 列表末尾
   - 同步记录到 `$DIARY_HOME/tag-changelog.md`

## Tag 列表格式（tags.md）

`$DIARY_HOME/tags.md` 使用简单列表：

```markdown
# Tag 体系

- 工作
- 会议
- 阅读
- 饮食
- 运动
- 社交
- 健康
- 财务
- 杂项
```

随使用自然增长，不需要维护层级关系。Tag 命名使用简短中文词语，保持直觉性。

## Tag 变迁日志格式（tag-changelog.md）

`$DIARY_HOME/tag-changelog.md` 记录每次 Tag 列表的变更：

```markdown
# Tag 变迁日志

## 2026-03-17

- **新增** `建模`
  - 触发记录: "今天第一次用 Blender 建了个低模角色"
- **新增** `烘焙`
  - 触发记录: "下午试着做了个戚风蛋糕"

## 2026-03-15

- **新增** `就医`
  - 触发记录: "下午去了趟口腔医院洗牙"
```

**追加规则**：如果当天的日期标题 `## YYYY-MM-DD` 已存在，将新条目追加到该标题下；不存在则在文件顶部（`# Tag 变迁日志` 之后）新建该日期段。最新日期始终在最上面。

## Tag 维护规则

1. **新增 Tag** 时必须记录：日期、新 Tag 名、触发该新增的原始记录
2. **不要删除 Tag**：即使暂时没有条目使用，保留以备将来
3. **避免同义 Tag**：新增前检查是否已有语义相近的 Tag
4. **Tag 粒度**：宁粗勿细，只有某类记录明显增多且现有 Tag 不够精确时才新增
